# Hiddify Addon Stack

> **Независимый community-проект.**
> Это независимый community overlay addon stack для Hiddify Manager 12.0.0.
> Проект не является аффилированным с проектом Hiddify, не одобрен им и официально не поддерживается.

---

## Что это такое

Этот проект расширяет Hiddify Manager 12.0.0 серверным управлением роутингом, управлением upstream relay-нодами, операционным инструментарием и опциональными модулями биллинга и защиты от шаринга — устанавливается как overlay поверх стандартной установки Hiddify Manager.

**Почему overlay, а не fork:**
Изначально проект рассматривался как fork Hiddify Manager 12.0.0. После выхода Hiddify 12.3.0 со значительными изменениями внутренней архитектуры поддержка полного fork стала нецелесообразной. Проект был переработан в overlay-установщик: файлы аддонов копируются в runtime панели при установке, базовая панель остаётся неизменной и обновляемой независимо, а откат восстанавливает оригинальные файлы из pre-install резервной копии.

**Опциональность модулей:**
Все три модуля аддонов логически независимы. Можно использовать только business/billing слой на single-node установке без включения routing или anti-share. Relay/cascade сценарий — для операторов, которым нужно управление upstream-нодами и серверная маршрутизация. Anti-share — опциональный слой защиты от шаринга аккаунтов.

---

## Совместимость

> ⚠️ **Этот релиз поддерживает только Hiddify Manager 12.0.0.**

Hiddify Manager 12.3.0 и новее **не поддерживаются** этим релизом. В upstream-проекте между 12.0.0 и 12.3.0 изменилась внутренняя архитектура и расположение файлов. Перенос addon-стека на новые версии Hiddify требует отдельной адаптации и проверки.

**Не устанавливайте этот релиз на Hiddify Manager 12.3.0+, если вы специально не занимаетесь портированием.**

## Почему такая архитектура

Этот стек аддонов разрабатывался под конкретный операционный сценарий: **локальная или региональная входная нода**, через которую подключаются пользователи, плюс **серверная маршрутизация**, которая решает куда идёт каждый поток трафика — без необходимости что-либо настраивать на устройстве пользователя.

Три практические проблемы определили эту архитектуру:

### Совместимость приложений при активном VPN-интерфейсе

Некоторые приложения и сайты детектируют, активен ли VPN-интерфейс на устройстве пользователя. Поведение разное: одни сервисы работают нормально если видимый IP принадлежит ожидаемому региону, другие ограничивают доступ независимо. На практике операторы наблюдают, что использование **локальной или региональной входной ноды** с **серверной маршрутизацией** может уменьшить количество ложных срабатываний и улучшить совместимость в некоторых сценариях. Пользователю не нужно вручную настраивать раздельное туннелирование — решения принимаются на сервере.

Это не гарантирует работу всех приложений. Результат зависит от конкретного метода детектирования.

### Учёт сетевого трафика

У некоторых интернет-провайдеров и мобильных операторов международный или межрегиональный трафик тарифицируется отдельно от локального/домашнего и может стоить дороже. Подключение пользователей к **локальному серверу** означает, что их трафик учитывается как локальный. Локальный сервер направляет через relay-ноду только те потоки, которым нужен внешний upstream, удерживая объём тарифицируемого международного трафика минимальным.

Этот подход позволяет управлять расходами на трафик без изменения пользовательского профиля подключения.

### Серверная маршрутизация вместо клиентской

Распространять сложные конфигурации маршрутизации на конечных пользователей ненадёжно — у клиентов могут быть устаревшие правила, неправильно настроенное раздельное туннелирование, или они просто не знают как это делается. Этот стек переносит решения на сервер: оператор один раз настраивает что идёт через прокси, что напрямую, что дропается. Пользователи подключаются с одним стандартным профилем и автоматически получают правильную маршрутизацию.

## Компоненты

| Компонент | Описание |
|---|---|
| **Routing addon** | Серверные правила роутинга, управление upstream-нодами (VLESS, Trojan, WireGuard), поддержка relay-ноды, health probe |
| **Anti-share addon** | Обнаружение шаринга аккаунтов по IP-скорингу, опциональное применение nftables |
| **Business addon** | Telegram-бот, тарифные планы, биллинговые хуки, интеграция оплаты |

Каждый компонент имеет собственный установщик, smoke test, манифест и скрипт отката. Их можно устанавливать независимо или вместе через общий wrapper.

## Что добавляет каждый модуль

### Роутинг и управление трафиком

Серверные правила маршрутизации, управляемые из интерфейса администратора.

- Выберите, какие домены, IP-диапазоны или CIDR-блоки маршрутизируются, проксируются или блокируются.
- Локальный/домашний трафик может идти напрямую; выбранные адреса перенаправляются через upstream relay-ноду.
- Роутинг применяется к Xray и Sing-box через `apply_configs.sh`.

**Источники правил** — правила организованы в источники. Каждый источник — список доменов или подсетей с одной политикой (`direct` / `upstream` / `block`). Источники добавляются тремя способами:

| Тип | Описание |
|---|---|
| **URL** | Удалённый список, загружаемый автоматически; поддерживает ручной re-import |
| **Файл** | Локальный файл на сервере (например `/opt/hiddify-manager/routing-lists/mylist.txt`) |
| **Текст** | Список, введённый вручную прямо в интерфейсе администратора |

Поддерживаемые форматы: список доменов, CIDR-подсети или смешанный. Каждая строка — одно правило. Комментарии (`#`, `//`) игнорируются. После изменений нажмите **Применить xray-router** в интерфейсе.

### Relay-нода и управление upstream

Один сервер может одновременно выступать хостом панели и routing relay:

- Добавление, редактирование и удаление upstream relay-нод из интерфейса (VLESS, Trojan, WireGuard).
- Переключение роутинга между upstream без правки конфигурационных файлов.
- Статус доступности upstream виден в панели (60-секундный probe записывает статус в БД и JSON-файл).

### Защита от шаринга аккаунтов (опционально)

- Движок IP-скоринга читает xray access log и считает уникальные IP источников на UUID пользователя.
- Настраиваемый порог на тарифный план пользователя.
- Опциональное применение nftables — по умолчанию отключено, dry-run режим при первой установке.

### Telegram bot инструментарий

- **Самообслуживание пользователей** — статус подписки, инструкции по платформам (Android, iPhone, Windows).
- **Действия администратора** — новые регистрации, запросы тарифов, платёжные события.
- Автоматическая выдача пробного доступа при регистрации через номер телефона.
- Inline-ссылка на подписку в сообщениях о статусе.
- **Прокси для исходящих запросов к Telegram API** — если Telegram Bot API недоступен напрямую, исходящие API-запросы бота можно направить через SOCKS5 или HTTP прокси. Встроенный xray-router (порт 20808) можно использовать как такой прокси. Настраивается в интерфейсе → Business → Telegram → *Прокси Telegram API*. На входящие webhook-запросы не влияет.

### Биллинг / коммерческий слой (опционально)

- Тарифные планы с настраиваемыми лимитами данных и сроком действия.
- Интеграция платёжного провайдера (YooKassa).
- Автоматические напоминания об окончании срока через Celery.

Этот модуль опционален. Routing, управление relay-нодами и операционный инструментарий работают независимо от биллингового слоя.

## Подготовка перед установкой

### Шаг 0 — Установка Hiddify Manager 12.0.0

```bash
sudo apt update && sudo apt upgrade -y
bash <(curl https://raw.githubusercontent.com/hiddify/Hiddify-Manager/refs/tags/v12.0.0/common/download.sh) "v12.0.0"
```

После установки откройте панель в браузере и **завершите мастер начальной настройки** (аккаунт администратора, домен, настройки прокси). Не запускайте установщик аддонов до тех пор, пока базовая панель не работает корректно.

### Шаг 1 — Добавить swap (рекомендуется для серверов с 1 ГБ ОЗУ)

```bash
fallocate -l 2G /swapfile
chmod 600 /swapfile
mkswap /swapfile
swapon /swapfile
echo '/swapfile none swap sw 0 0' >> /etc/fstab
```

### Шаг 2 — Клонировать репозиторий

```bash
git clone https://github.com/daviddt369/hiddify-billing-router-guard.git
cd hiddify-billing-router-guard
```

## Чистая установка (полный стек)

```bash
sudo bash release/clean-install-full-stack.sh
```

Устанавливает все три аддона по порядку:

```text
business → routing → antishare
```

Каждый этап выполняет preflight-проверки, создаёт резервные копии, копирует файлы, применяет миграции БД, перезапускает сервисы и проверяет smoke-тестами. При сбое откатывается автоматически. Ожидаемое время: 10–15 минут.

Подробнее: [INSTALL.ru.md](INSTALL.ru.md)

## Шаги после установки

### Настройка Telegram-бота

1. Создайте бота через [@BotFather](https://t.me/BotFather).
2. Интерфейс администратора → **Business → Telegram** → введите токен и сохраните.
3. Отправьте боту команду активации:

   ```text
   /start admin_<UUID_АДМИНИСТРАТОРА>
   ```

   Точная команда сохранена на сервере:

   ```bash
   cat /opt/hiddify-manager/business-addon-secrets/telegram-owner-activation.txt
   ```

### Настройка upstream-ноды (опционально)

1. Интерфейс администратора → **Business → Routing** → добавьте upstream-ноду (VLESS, Trojan или WireGuard).
2. Включите роутинг и сохраните.
3. Примените конфигурацию для активации изменений в работающем прокси-ядре:

   ```bash
   sudo bash /opt/hiddify-manager/apply_configs.sh
   ```

### Исправление балансировщика proxy-stats (если установлен hiddify-cli)

```bash
sudo bash release/service-tools/fix-hiddify-cli-balancer.sh
```

## Ожидаемые предупреждения (не ошибки)

- **"Telegram bot token is not configured"** — появляется в логах при каждом запуске панели до настройки токена. Штатное поведение.
- **"xray-router inactive" / "upstream not reachable"** — появляется до тех пор, пока не настроена хотя бы одна upstream-нода в Business → Routing. Штатное поведение.
