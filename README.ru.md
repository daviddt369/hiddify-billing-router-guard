# Hiddify Addon Stack

> **Независимый community-проект.** Этот проект является независимым community addon stack для Hiddify Manager. Он не является аффилированным с проектом Hiddify, не одобрен им и официально не поддерживается, если явно не указано обратное.

Overlay-набор аддонов для [Hiddify Manager](https://github.com/hiddify/HiddifyPanel) версии 12.0.0, который добавляет серверное управление роутингом, управление upstream/relay-нодами, операционный инструментарий и опциональные расширения управления пользователями — в виде overlay-установщика поверх стандартной установки Hiddify Manager, без поддержки форка upstream Hiddify репозитория.

---

## Что добавляет этот проект

### 1. Роутинг и управление трафиком

Серверные правила маршрутизации, управляемые из интерфейса администратора панели. Полезно для операторов, чьи пользователи не настраивают клиентский роутинг и которым нужны разумные серверные настройки по умолчанию.

- Выберите, какие домены, IP-диапазоны или CIDR-блоки маршрутизируются, проксируются или блокируются — прямо из панели администратора.
- Локальный/внутренний трафик может идти напрямую; выбранные адреса перенаправляются через upstream relay-ноду.
- Роутинг автоматически применяется к Xray и Sing-box через `apply_configs.sh`.

**Источники правил** — правила роутинга организованы в источники. Каждый источник — это список доменов или подсетей с одной политикой (прямо / upstream / block). Источники добавляются тремя способами:

| Тип | Описание |
|---|---|
| **URL** | Удалённый список, загружаемый автоматически; поддерживает ручной и автоматический re-import |
| **Файл** | Локальный файл на сервере (например `/opt/hiddify-manager/routing-lists/mylist.txt`) |
| **Текст** | Список, введённый вручную прямо в интерфейсе администратора |

Поддерживаемые форматы: список доменов, список CIDR-подсетей или смешанный. Каждая строка — одно правило. Комментарии (`#`, `//`) игнорируются. После добавления или обновления любого источника нажмите **Применить xray-router** в интерфейсе для активации изменений.

### 2. Relay-нода и управление upstream

Один сервер может одновременно выступать хостом панели и routing relay:

- Добавление, редактирование и удаление upstream relay-нод из интерфейса администратора (VLESS, Trojan, WireGuard).
- Переключение роутинга между upstream без правки конфигурационных файлов.
- Статус доступности upstream виден в панели администратора и проверяется автоматическим probe.

### 3. Операционная безопасность

- **Скрипт чистой установки** — устанавливает все аддоны в правильном порядке с preflight-проверками и smoke-тестами на каждом этапе.
- **Скрипт обновления** — переприменяет файлы аддонов поверх обновлённой панели без затрагивания routing и antishare.
- **Smoke-тесты** — проверяют каждый аддон после установки или обновления.
- **Скрипты отката** — поаддонный и полный откат из pre-install резервных копий файлов.
- **Routing health probe** — таймер systemd каждые 60 секунд; оповещает администратора через Telegram при недоступности upstream.

### 4. Защита от шаринга аккаунтов

Опциональное обнаружение и применение санкций за шаринг аккаунтов, более актуальное для коммерческих или community-операторов:

- Движок IP-скоринга читает xray access log и считает уникальные IP источников на UUID пользователя.
- Настраиваемый порог на тарифный план пользователя.
- Опциональное применение nftables (по умолчанию отключено; dry-run режим при первой установке).

### 5. Telegram bot инструментарий

- **Самообслуживание пользователей** — статус подписки, инструкции по настройке для каждой платформы (Android, iPhone, Windows), контакт поддержки.
- **Действия администратора** — новые регистрации, запросы на тариф, платёжные события, уведомления о недоступности upstream.
- Автоматическая выдача пробного доступа при регистрации через номер телефона.
- Inline-ссылка на подписку в сообщениях о статусе.

### 6. Биллинг / коммерческий слой (опционально)

Опциональный коммерческий слой подписок для операторов, оказывающих платные или community VPN-услуги:

- Тарифные планы с настраиваемыми лимитами данных и сроком действия.
- Интеграция платёжного провайдера (YooKassa).
- Автоматические напоминания об окончании срока через Celery.

Этот компонент опционален. Роутинг, управление relay-нодами и операционный инструментарий работают независимо от биллингового слоя.

## Почему такая архитектура

Этот стек аддонов разрабатывался под конкретный операционный сценарий: **локальная точка входа**, через которую подключаются пользователи, плюс **серверная маршрутизация**, которая решает куда идёт каждый поток трафика — без необходимости что-либо настраивать на устройстве пользователя.

Три практические проблемы определили эту архитектуру:

### Совместимость приложений при активном VPN-интерфейсе

Некоторые приложения и сайты детектируют, активен ли VPN-интерфейс на устройстве пользователя. Поведение разное: одни сервисы работают нормально если видимый IP-адрес принадлежит ожидаемому региону, другие ограничивают доступ независимо от IP. На практике операторы наблюдают меньше проблем совместимости когда:

- пользователь подключается к **локальному или региональному серверу входа**, а не напрямую к зарубежному upstream
- **серверная маршрутизация** удерживает выбранные потоки трафика локально, и видимый исходящий IP для этих потоков остаётся региональным
- пользователю не нужно вручную настраивать раздельное туннелирование — решения о маршрутизации принимаются на сервере

Это не гарантирует совместимость с каждым приложением или сервисом. Результат зависит от конкретного метода детектирования.

### Учёт сетевого трафика

У некоторых интернет-провайдеров и мобильных операторов **международный или межрегиональный трафик** тарифицируется отдельно от домашнего/локального и может стоить существенно дороже за гигабайт. Подключение пользователей к **локальному серверу** означает, что их трафик учитывается как локальный. Локальный сервер затем направляет через relay-ноду только те потоки, которым действительно нужен внешний upstream, удерживая объём тарифицируемого международного трафика минимальным.

Этот подход позволяет операторам управлять расходами на трафик без изменения пользовательского профиля подключения.

### Серверная маршрутизация вместо клиентской

Распространять сложные конфигурации маршрутизации на конечных пользователей ненадёжно — у клиентов могут быть устаревшие наборы правил, неправильно настроенное раздельное туннелирование, или они просто не знают как это делается. Этот стек переносит решения о маршрутизации на сервер: оператор один раз настраивает что идёт через прокси, что напрямую, что дропается. Пользователи подключаются с одним стандартным профилем и автоматически получают правильную маршрутизацию.

## Подготовка перед установкой

### 0. Установка Hiddify Manager 12.0.0

Этот стек аддонов требует **строго версии 12.0.0**. Другие версии не поддерживаются.

```bash
sudo apt update && sudo apt upgrade -y
bash <(curl https://raw.githubusercontent.com/hiddify/Hiddify-Manager/refs/tags/v12.0.0/common/download.sh) "v12.0.0"
```

Дождитесь полного запуска панели перед продолжением.

### 0а. Пройдите начальную настройку панели

После установки откройте панель в браузере и завершите мастер начальной конфигурации (аккаунт администратора, домен, настройки прокси). Установщик аддонов требует, чтобы панель была полностью настроена и её сервисы были активны.

**Не запускайте установщик аддонов до тех пор, пока базовая панель не работает корректно.**

### 1. Добавить swap (рекомендуется для серверов с 1 ГБ ОЗУ)

```bash
fallocate -l 2G /swapfile
chmod 600 /swapfile
mkswap /swapfile
swapon /swapfile
echo '/swapfile none swap sw 0 0' >> /etc/fstab
```

### 2. Клонировать репозиторий

```bash
git clone https://github.com/daviddt369/hiddify-billing-router-guard.git
cd hiddify-billing-router-guard
```

## Чистая установка

```bash
sudo bash release/clean-install-full-stack.sh
```

Устанавливает все три аддона по порядку (business → routing → antishare) с preflight-проверками, миграциями БД, smoke-тестами и автоматическим откатом при сбое. Ожидаемое время: 10–15 минут.

Подробнее: [INSTALL.ru.md](INSTALL.ru.md)

## Ручные шаги после установки

### Настройка Telegram-бота

1. Создайте бота через [@BotFather](https://t.me/BotFather).
2. Интерфейс администратора → **Business → Telegram** → введите токен и сохраните.
3. Отправьте боту команду активации:

   ```
   /start admin_<UUID_АДМИНИСТРАТОРА>
   ```

   Точная команда сохранена на сервере:

   ```bash
   cat /opt/hiddify-manager/business-addon-secrets/telegram-owner-activation.txt
   ```

### Настройка upstream-ноды и применение конфигурации (опционально)

Для включения маршрутизации трафика через внешнюю relay-ноду:

1. Интерфейс администратора → **Business → Routing** → добавьте upstream-ноду (формат VLESS, Trojan или WireGuard).
2. Включите роутинг в том же разделе и сохраните.
3. **Примените конфигурацию** — обязательный шаг для активации роутинга в Xray/Sing-box:

   ```bash
   sudo bash /opt/hiddify-manager/apply_configs.sh
   ```

   Без этого шага изменения сохраняются в БД, но не активны в работающем прокси-ядре.

### Исправление балансировщика proxy-stats (если установлен hiddify-cli)

```bash
sudo bash release/service-tools/fix-hiddify-cli-balancer.sh
```

## Ожидаемые предупреждения (не ошибки)

- **"Telegram bot token is not configured"** — появляется в логах при каждом запуске панели до настройки токена в интерфейсе администратора.
- **"xray-router inactive" / "upstream not reachable"** — появляется до тех пор, пока не настроена хотя бы одна upstream-нода в разделе Business → Routing.
